<p>We are finally ready to see what makes the shell such a powerful programming environment.

<p>Not only will writing shell scripts make your work faster– you won't have to retype the same commands over and over

<p>What if we want to process many files in a single pipeline?
For example, if we want to sort our <code>.pdb</code> files by length, we would type:</p>

<p>because <code>wc -l</code> lists the number of lines in the files

but then it would only ever sort a list of <code>.pdb</code> files in the current directory.

we need a way to get all those names into the script.

which means,
'All of the command-line arguments to the shell script'.
We also should put <code>$@</code> inside double-quotes
to handle the case of arguments containing spaces
(<code>"$@"</code> is special syntax and is equivalent to <code>"$1"</code> <code>"$2"</code> ...).</p>

<h2><a id="user-content-why-record-commands-in-the-history-before-running-them" class="anchor" aria-hidden="true" href="#why-record-commands-in-the-history-before-running-them"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Why Record Commands in the History Before Running Them?</h2>
<p>If you run the command:</p>
<div class="snippet-clipboard-content position-relative" data-snippet-clipboard-copy-content="$ history | tail -n 5 &gt; recent.sh
"><pre><code>$ history | tail -n 5 &gt; recent.sh
</code></pre></div>
<p>{: .language-bash}</p>
<p>the last command in the file is the <code>history</code> command itself, i.e.,
the shell has added <code>history</code> to the command log before actually
running it. In fact, the shell <em>always</em> adds commands to the log
before running them. Why do you think it does this?</p>
<blockquote>
<h2><a id="user-content-solution-1" class="anchor" aria-hidden="true" href="#solution-1"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Solution</h2>
<p>If a command causes something to crash or hang, it might be useful
to know what that command was, in order to investigate the problem.
Were the command only be recorded after running it, we would not
have a record of the last command run in the event of a crash.
{: .solution}
{: .challenge}</p>
</blockquote>
</blockquote>
<p>In practice, most people develop shell scripts by running commands at the shell prompt a few times
to make sure they're doing the right thing,
then saving them in a file for re-use.
This style of work allows people to recycle
what they discover about their data and their workflow with one call to <code>history</code>
and a bit of editing to clean up the output
and save it as a shell script.</p>
<h2><a id="user-content-nelles-pipeline-creating-a-script" class="anchor" aria-hidden="true" href="#nelles-pipeline-creating-a-script"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Nelle's Pipeline: Creating a Script</h2>
<p>Nelle's supervisor insisted that all her analytics must be reproducible. The easiest way to capture all the steps is in a script.</p>
<p>First we return to Nelle's data directory:</p>
<div class="snippet-clipboard-content position-relative" data-snippet-clipboard-copy-content="$ cd ../north-pacific-gyre/2012-07-03/
"><pre><code>$ cd ../north-pacific-gyre/2012-07-03/
</code></pre></div>
<p>{: .language-bash}</p>
<p>She runs the editor and writes the following:</p>
<div class="snippet-clipboard-content position-relative" data-snippet-clipboard-copy-content="# Calculate stats for data files.
for datafile in &quot;$@&quot;
do
    echo $datafile
    bash goostats.sh $datafile stats-$datafile
done
"><pre><code># Calculate stats for data files.
for datafile in "$@"
do
    echo $datafile
    bash goostats.sh $datafile stats-$datafile
done
</code></pre></div>
<p>{: .language-bash}</p>
<p>She saves this in a file called <code>do-stats.sh</code>
so that she can now re-do the first stage of her analysis by typing:</p>
<div class="snippet-clipboard-content position-relative" data-snippet-clipboard-copy-content="$ bash do-stats.sh NENE*[AB].txt
"><pre><code>$ bash do-stats.sh NENE*[AB].txt
</code></pre></div>
<p>{: .language-bash}</p>
<p>She can also do this:</p>
<div class="snippet-clipboard-content position-relative" data-snippet-clipboard-copy-content="$ bash do-stats.sh NENE*[AB].txt | wc -l
"><pre><code>$ bash do-stats.sh NENE*[AB].txt | wc -l
</code></pre></div>
<p>{: .language-bash}</p>
<p>so that the output is just the number of files processed
rather than the names of the files that were processed.</p>
<p>One thing to note about Nelle's script is that
it lets the person running it decide what files to process.
She could have written it as:</p>
<div class="snippet-clipboard-content position-relative" data-snippet-clipboard-copy-content="# Calculate stats for Site A and Site B data files.
for datafile in NENE*[AB].txt
do
    echo $datafile
    bash goostats.sh $datafile stats-$datafile
done
"><pre><code># Calculate stats for Site A and Site B data files.
for datafile in NENE*[AB].txt
do
    echo $datafile
    bash goostats.sh $datafile stats-$datafile
done
</code></pre></div>
<p>{: .language-bash}</p>
<p>The advantage is that this always selects the right files:
she doesn't have to remember to exclude the 'Z' files.
The disadvantage is that it <em>always</em> selects just those files --- she can't run it on all files
(including the 'Z' files),
or on the 'G' or 'H' files her colleagues in Antarctica are producing,
without editing the script.
If she wanted to be more adventurous,
she could modify her script to check for command-line arguments,
and use <code>NENE*[AB].txt</code> if none were provided.
Of course, this introduces another tradeoff between flexibility and complexity.</p>
<blockquote>
<h2><a id="user-content-variables-in-shell-scripts" class="anchor" aria-hidden="true" href="#variables-in-shell-scripts"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Variables in Shell Scripts</h2>
<p>In the <code>molecules</code> directory, imagine you have a shell script called <code>script.sh</code> containing the
following commands:</p>
<div class="snippet-clipboard-content position-relative" data-snippet-clipboard-copy-content="head -n $2 $1
tail -n $3 $1
"><pre><code>head -n $2 $1
tail -n $3 $1
</code></pre></div>
<p>{: .language-bash}</p>
<p>While you are in the <code>molecules</code> directory, you type the following command:</p>

</code></pre></div>

<blockquote>
<h2><a id="user-content-solution-2" class="anchor" aria-hidden="true" href="#solution-2"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Solution</h2>

<p>The special variables $1, $2 and $3 represent the command line arguments given to the
script, such that the commands run are:</p>
<div class="snippet-clipboard-content position-relative" data-snippet-clipboard-copy-content="$ head -n 1 cubane.pdb ethane.pdb octane.pdb pentane.pdb propane.pdb
$ tail -n 1 cubane.pdb ethane.pdb octane.pdb pentane.pdb propane.pdb
"><pre><code>$ head -n 1 cubane.pdb ethane.pdb octane.pdb pentane.pdb propane.pdb

As such, the first argument to the script is <code>'*.pdb'</code> which gets expanded within the

{: .solution}
{: .challenge}</p>

<h2><a id="user-content-find-the-longest-file-with-a-given-extension" class="anchor" aria-hidden="true" href="#find-the-longest-file-with-a-given-extension"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Find the Longest File With a Given Extension</h2>
<p>Write a shell script called <code>longest.sh</code> that takes the name of a

<div class="snippet-clipboard-content position-relative" data-snippet-clipboard-copy-content="$ bash longest.sh /tmp/data pdb
"><pre><code>$ bash longest.sh /tmp/data pdb
</code></pre></div>
<p>{: .language-bash}</p>
<p>would print the name of the <code>.pdb</code> file in <code>/tmp/data</code> that has
the most lines.</p>
<blockquote>
<h2><a id="user-content-solution-3" class="anchor" aria-hidden="true" href="#solution-3"><svg class="octicon octicon-link" viewBox="0 0 16 16" version="1.1" width="16" height="16" aria-hidden="true"><path fill-rule="evenodd" d="M7.775 3.275a.75.75 0 001.06 1.06l1.25-1.25a2 2 0 112.83 2.83l-2.5 2.5a2 2 0 01-2.83 0 .75.75 0 00-1.06 1.06 3.5 3.5 0 004.95 0l2.5-2.5a3.5 3.5 0 00-4.95-4.95l-1.25 1.25zm-4.69 9.64a2 2 0 010-2.83l2.5-2.5a2 2 0 012.83 0 .75.75 0 001.06-1.06 3.5 3.5 0 00-4.95 0l-2.5 2.5a3.5 3.5 0 004.95 4.95l1.25-1.25a.75.75 0 00-1.06-1.06l-1.25 1.25a2 2 0 01-2.83 0z"></path></svg></a>Solution</h2>
<div class="snippet-clipboard-content position-relative" data-snippet-clipboard-copy-content="# Shell script which takes two arguments:
#    1. a directory name
#    2. a file extension
# and prints the name of the file in that directory
# with the most lines which matches the file extension.

wc -l $1/*.$2 | sort -n | tail -n 2 | head -n 1
"><pre><code># Shell script which takes two arguments:
#    1. a directory name
#    2. a file extension
# and prints the name of the file in that directory
# with the most lines which matches the file extension.

wc -l $1/*.$2 | sort -n | tail -n 2 | head -n 1
</code></pre></div>
<p>{: .source}</p>
<p>The first part of the pipeline, <code>wc -l $1/*.$2 | sort -n</code>, counts
the lines in each file and sorts them numerically (largest last). When
there's more than one file, <code>wc</code> also outputs a final summary line,
giving the total number of lines across <em>all</em> files.  We use <code>tail -n 2 | head -n 1</code> to throw away this last line.</p>
<p>With <code>wc -l $1/*.$2 | sort -n | tail -n 1</code> we'll see the final summary
line: we can build our pipeline up in pieces to be sure we understand
the output.</p>
